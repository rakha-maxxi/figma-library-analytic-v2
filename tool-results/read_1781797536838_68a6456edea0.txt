     1→# Product Requirements Document (PRD)
     2→
     3→# Design System Component Usage Tracker
     4→
     5→## 1. Overview
     6→
     7→### 1.1 Product Name
     8→
     9→**Design System Component Usage Tracker**
    10→
    11→Nama alternatif:
    12→
    13→- Component Usage Tracker
    14→- Design System Analytics Tracker
    15→- UI Kit Usage Tracker
    16→- Component Adoption Tracker
    17→
    18→---
    19→
    20→### 1.2 Background
    21→
    22→Design System Designer / Design Ops bertanggung jawab menyediakan UI Kit, component library, dan design system asset agar tim Product Designer dapat membuat desain UI/UX lebih cepat, konsisten, dan scalable.
    23→
    24→Namun, setelah component dibuat dan digunakan oleh tim product designer, sering muncul gap visibility:
    25→
    26→- Design Ops tidak tahu component tertentu dipakai di file mana saja.
    27→- Sulit mengetahui apakah component library benar-benar diadopsi oleh tim.
    28→- Sulit mengetahui component mana yang aktif, jarang digunakan, atau tidak pernah digunakan.
    29→- Sulit melihat perubahan usage component dari waktu ke waktu.
    30→- Sulit membuat keputusan cleanup, improvement, atau deprecation berdasarkan data.
    31→- Untuk beberapa Figma plan, fitur analytics bawaan seperti Figma Library Analytics tidak tersedia.
    32→
    33→Produk ini dibuat sebagai dashboard internal untuk membantu Design Ops melacak penggunaan component dari satu atau lebih UI Kit/design system source file terhadap file-file Figma product/design yang sudah didaftarkan.
    34→
    35→---
    36→
    37→### 1.3 Problem Statement
    38→
    39→Sebagai Design Ops, user membutuhkan cara cepat dan jelas untuk menjawab pertanyaan:
    40→
    41→> “Component dari design system ini dipakai di file mana saja, seberapa banyak, dan bagaimana perubahannya dari waktu ke waktu?”
    42→
    43→Tanpa tool ini, Design Ops harus membuka file product satu per satu untuk mengecek penggunaan component. Proses ini memakan waktu, rawan tidak akurat, sulit diulang, dan sulit diskalakan ketika jumlah file product semakin banyak.
    44→
    45→---
    46→
    47→### 1.4 Product Goal
    48→
    49→Membuat dashboard internal untuk melacak, menganalisis, dan memonitor penggunaan component dari design system / UI Kit source file pada file-file Figma yang terdaftar.
    50→
    51→Produk ini akan membantu user melihat:
    52→
    53→- daftar component dari source UI Kit
    54→- total usage setiap component
    55→- file mana saja yang menggunakan component tertentu
    56→- jumlah instance component per file
    57→- status component: active, low usage, unused
    58→- usage change dari scan sebelumnya
    59→- file-level design system adoption
    60→- scan history
    61→- basic governance insights
    62→

    83→
    84→#### Objective 3: Membantu design system governance berbasis data
    85→

    87→
    88→- User dapat mengidentifikasi unused components.

    93→#### Objective 4: Menyediakan alternatif scoped analytics untuk tim yang belum memiliki full library analytics bawaan

    98→- User dapat melihat component usage berdasarkan scan history.
    99→- User dapat memahami scope dan limitation data secara jelas.

   176→## 2. Scope
   177→
   178→### 2.1 In-Scope
   179→
   180→#### Source UI Kit / Design System File Management
   181→
   182→- User dapat mendaftarkan satu source UI Kit/design system file.
   183→- User dapat melihat daftar component dari source file.
   184→- User dapat mengganti source file jika diperlukan.
   185→- User dapat refresh component inventory.
   186→
   187→#### Registered Consumer Files
   188→
   189→- User dapat mendaftarkan file-file Figma yang ingin dilacak.
   190→- User dapat melihat daftar registered files.
   191→- User dapat menghapus atau disable file dari tracking.
   192→- User dapat melihat status scan setiap file.
   193→
   194→#### Component Usage Tracking
   195→
   196→- Sistem dapat menampilkan total usage setiap component.
   197→- Sistem dapat menampilkan jumlah file yang menggunakan component.
   198→- Sistem dapat menampilkan component status:
   199→  - Active
   200→  - Low Usage
   201→  - Unused
   202→  - Not Scanned
   203→
   204→- User dapat mencari, memfilter, dan mengurutkan component.
   205→
   206→#### Component Detail
   207→
   208→- User dapat membuka detail component.
   209→- User dapat melihat file mana saja yang menggunakan component tersebut.
   210→- User dapat melihat jumlah instance component per file.
   211→- User dapat melihat lokasi instance seperti page/frame jika tersedia.
   212→- User dapat membuka file atau node terkait di Figma jika memungkinkan.
   213→
   214→#### File-Level Usage
   215→
   216→- User dapat melihat ringkasan usage per registered file.
   217→- User dapat melihat total design system instances dalam file.
   218→- User dapat melihat unique components yang digunakan dalam file.
   219→- User dapat membuka detail file untuk melihat component apa saja yang dipakai.
   220→
   221→#### Scan Management
   222→
   223→- User dapat menjalankan scan terhadap semua registered files.
   224→- User dapat menjalankan rescan pada satu file tertentu.
   225→- User dapat melihat status scan:
   226→  - Pending
   227→  - Running
   228→  - Success
   229→  - Failed
   230→  - Paused
   231→
   232→- User dapat melihat scan history.
   233→- User dapat retry failed scan.
   234→
   235→#### Snapshot & Change Detection
   236→
   237→- Sistem menyimpan hasil scan sebagai snapshot.
   238→- Sistem dapat membandingkan hasil scan terbaru dengan scan sebelumnya.
   239→- Sistem dapat menampilkan perubahan usage:
   240→  - Newly Used
   241→  - Increased
   242→  - Decreased
   243→  - Removed
   244→
   245→- Sistem dapat menampilkan recent changes.
   246→
   247→#### Insights
   248→
   249→- Sistem dapat menampilkan unused components.
   250→- Sistem dapat menampilkan low usage components.
   251→- Sistem dapat menampilkan most used components.
   252→- Sistem dapat menampilkan stale files.
   253→- Sistem dapat menampilkan failed scans.
   254→
   255→#### UI & Experience
   256→
   257→- UI mengambil inspirasi dari Figma Library Analytics untuk domain analytics.
   258→- UX mengambil inspirasi dari Linear untuk interaction, clarity, density, dan speed.
   259→- UI dapat dibangun menggunakan shadcn/ui sebagai design library.
   260→- Dashboard harus clean, compact, dan data-first.

   392→## 3. User Stories / Use Cases
   393→
   394→### 3.1 Register Source UI Kit
   395→
   396→As a Design Ops, I want to register a source UI Kit file so that the system can identify the components that should be tracked.
   397→
   398→#### Happy Path
   399→
   400→1. User opens onboarding or settings.
   401→2. User pastes source UI Kit Figma file link.
   402→3. System validates the link.
   403→4. System displays source file name.
   404→5. System imports component list.
   405→6. User sees confirmation that source UI Kit is connected.
   406→
   407→#### Edge Cases
   408→
   409→- File link is invalid.
   410→- File cannot be accessed.
   411→- File does not contain components.
   412→- User registers the wrong file.
   413→- Source UI Kit was previously registered.
   414→
   415→---
   416→
   417→### 3.2 Register Consumer Files
   418→
   419→As a Design Ops, I want to register product/design files so that the system can scan them for design system component usage.
   420→
   421→#### Happy Path
   422→
   423→1. User opens Files page.
   424→2. User clicks “Add files”.
   425→3. User pastes one or multiple Figma file links.
   426→4. System validates each file.
   427→5. System adds valid files to the registered files list.
   428→6. User sees file name, status, and scan state.
   429→
   430→#### Edge Cases
   431→
   432→- One or more links are invalid.
   433→- File already exists in registered list.
   434→- File cannot be accessed.
   435→- User adds unsupported file type.
   436→- User pastes mixed valid and invalid links.
   437→
   438→---
   439→
   440→### 3.3 Scan Registered Files
   441→
   442→As a Design Ops, I want to scan registered files so that I can see which source components are used in those files.
   443→
   444→#### Happy Path
   445→
   446→1. User clicks “Scan all”.
   447→2. System creates scan job.
   448→3. System shows scan progress.
   449→4. Each file moves through pending, running, and success state.
   450→5. Usage result is saved.
   451→6. Dashboard updates component and file usage summary.
   452→
   453→#### Edge Cases
   454→
   455→- One file fails but others succeed.
   456→- Scan is paused due to rate limit.
   457→- User closes dashboard during scan.
   458→- File was deleted or moved.
   459→- User no longer has access to a file.
   460→- Scan returns zero usage for a file.
   461→
   462→---
   463→
   464→### 3.4 View Component Usage List
   465→
   466→As a Design Ops, I want to see all source components and their usage count so that I can identify active, low usage, and unused components.
   467→
   468→#### Happy Path
   469→
   470→1. User opens Components page.
   471→2. User sees component table.
   472→3. Table shows component name, total instances, files used, status, and last seen.
   473→4. User sorts by total instances.
   474→5. User filters by status.
   475→6. User searches a component name.
   476→
   477→#### Edge Cases
   478→
   479→- No scan has been run yet.
   480→- Component list is empty.
   481→- Usage data is outdated.
   482→- Search returns no result.
   483→- Some components have duplicate or similar names.
   484→
   485→---
   486→
   487→### 3.5 View Component Detail
   488→
   489→As a Design Ops, I want to open a component detail page so that I can see where that component is being used.
   490→
   491→#### Happy Path
   492→
   493→1. User clicks Component A.
   494→2. System opens component detail page or drawer.
   495→3. User sees total instances and files used.
   496→4. User sees list of files using Component A.
   497→5. User sees instance count per file.
   498→6. User can open related Figma file or node.
   499→
   500→#### Edge Cases
   501→
   502→- Component has zero usage.
   503→- Component was deleted or renamed in source UI Kit.
   504→- Component exists but has no scan history.
   505→- Some file data is stale.
   506→- Some instance links cannot be generated.
   507→
   508→---
   509→
   510→### 3.6 View File Usage Summary
   511→
   512→As a Design Ops, I want to see usage summary per file so that I can evaluate design system adoption by file.
   513→
   514→#### Happy Path
   515→
   516→1. User opens Files page.
   517→2. User sees all registered files.
   518→3. Each file shows total design system instances, unique components used, status, and last scanned.
   519→4. User opens a file detail.
   520→5. User sees which source components are used in that file.
   521→
   522→#### Edge Cases
   523→
   524→- File has never been scanned.
   525→- File scan failed.
   526→- File has zero design system component usage.
   527→- File has stale scan data.
   528→- File is no longer accessible.
   529→
   530→---
   531→
   532→### 3.7 Detect Usage Changes
   533→
   534→As a Design Ops, I want to see changes between scans so that I can understand whether component usage is increasing, decreasing, newly added, or removed.
   535→
   536→#### Happy Path
   537→
   538→1. User runs a second scan after previous scan exists.
   539→2. System compares latest scan with previous scan.
   540→3. System detects changes.
   541→4. User opens Recent Changes.
   542→5. User sees component, file, previous count, current count, and change type.
   543→
   544→#### Edge Cases
   545→
   546→- There is no previous scan.
   547→- A file was skipped.
   548→- Component was renamed.
   549→- Component was removed from source UI Kit.
   550→- File failed during latest scan.
   551→- Usage count changes because instances were detached or deleted.
   552→
   553→---
   554→
   555→### 3.8 Identify Unused Components
   556→
   557→As a Design Ops, I want to see unused components so that I can review candidates for cleanup or deprecation.
   558→
   559→#### Happy Path
   560→
   561→1. User opens Insights page.
   562→2. User selects “Unused components”.
   563→3. System shows components with zero usage across registered files.
   564→4. User reviews component list.
   565→5. User decides whether component needs follow-up.
   566→
   567→#### Edge Cases
   568→
   569→- Component is intentionally unused but still needed.
   570→- Component is new and not adopted yet.
   571→- Registered file coverage is incomplete.
   572→- Usage exists in unregistered files but not visible to the system.
   573→

   576→## 4. Functional Requirements
   577→
   578→### 4.1 Onboarding
   579→
   580→The product must provide an onboarding flow for first-time setup.
   581→
   582→Expected behavior:
   583→
   584→- User is guided to connect Figma access.
   585→- User is guided to register source UI Kit.
   586→- User is guided to register consumer files.
   587→- User is informed that analytics are based only on registered files.
   588→- User is informed that scan result is snapshot-based, not real-time.
   589→
   590→---
   591→
   592→### 4.2 Source UI Kit Registration
   593→
   594→The product must allow user to register a source UI Kit file.
   595→
   596→Expected behavior:
   597→
   598→- User can paste a Figma file link.
   599→- System validates the link format.
   600→- System identifies the file as source UI Kit.
   601→- System displays file name after successful registration.
   602→- System can refresh component list from source UI Kit.
   603→- User can replace the registered source UI Kit file if needed.
   604→
   605→---
   606→
   607→### 4.3 Component Inventory
   608→
   609→The product must display component inventory from source UI Kit.
   610→
   611→Expected behavior:
   612→
   613→- Components are listed in a table.
   614→- Component name is visible.
   615→- Component set or grouping is visible if available.
   616→- Component status is visible.
   617→- Component usage summary is visible after scan.
   618→- User can search components.
   619→- User can sort components by usage.
   620→- User can filter components by status.
   621→
   622→Recommended statuses:
   623→
   624→- Active
   625→- Low Usage
   626→- Unused
   627→- Not Scanned
   628→- Deprecated Candidate
   629→
   630→---
   631→
   632→### 4.4 Registered Files Management
   633→
   634→The product must allow user to manage tracked files.
   635→
   636→Expected behavior:
   637→
   638→- User can add one or multiple Figma file links.
   639→- User can remove files from tracking.
   640→- User can disable files from future scans.
   641→- User can see file name, scan status, and last scanned date.
   642→- User can manually rescan a single file.
   643→- User can open a file in Figma.
   644→
   645→Recommended file statuses:
   646→
   647→- Not Scanned
   648→- Healthy
   649→- Low Adoption
   650→- Zero Usage
   651→- Failed
   652→- Stale
   653→- Disabled
   654→
   655→---
   656→
   657→### 4.5 Scan Management
   658→
   659→The product must allow user to start and monitor scans.
   660→
   661→Expected behavior:
   662→
   663→- User can trigger scan all.
   664→- User can trigger scan per file.
   665→- System displays scan progress.
   666→- System displays running, success, failed, and paused states.
   667→- System stores each scan as history.
   668→- System preserves previous scan history.
   669→- System does not erase historical data when latest scan finishes.
   670→- System preserves previous successful data if latest scan fails.
   671→
   672→---
   673→
   674→### 4.6 Component Usage Dashboard
   675→
   676→The product must show component usage across registered files.
   677→
   678→Expected behavior:
   679→
   680→- User can see total instances per component.
   681→- User can see number of files using each component.
   682→- User can see last seen timestamp.
   683→- User can open component detail.
   684→- User can identify unused components.
   685→- User can identify low usage components.
   686→- User can identify most used components.
   687→
   688→---
   689→
   690→### 4.7 Component Detail
   691→
   692→The product must show detailed usage for a selected component.
   693→
   694→Expected behavior:
   695→
   696→- Component detail shows total instances.
   697→- Component detail shows total files used.
   698→- Component detail shows usage by file.

  1285→
  1286→### 7.1 Dependencies
  1287→
  1288→The product depends on:
  1289→
  1290→#### Figma Access
  1291→
  1292→- User must have access to source UI Kit file.
  1293→- User must have access to registered product files.

  1297→
  1298→- Component usage can only be detected if component instances remain linked to the source component.

  1300→- Deleted files or inaccessible files cannot be scanned.
  1301→

  1303→
  1304→- Product analytics depends on the list of registered files.
  1305→- Files not registered are not included in usage results.
  1306→- User must maintain tracking coverage.
  1307→
  1308→#### Scan History
  1309→
  1310→- Trend and change detection depend on multiple scans.
  1311→- First scan cannot produce increase/decrease trend.
  1312→- Change detection is based on scan interval, not exact edit timestamp.
  1313→
  1314→#### UI Component Library
  1315→
  1316→- UI can use shadcn/ui as base component library.
  1317→- Tables, badges, dialogs, drawers, tabs, and forms should follow consistent patterns.

  1322→
  1323→- The primary user is Design Ops or Design System Designer.
  1324→- The first version is internal-use only.
  1325→- The MVP tracks one main source UI Kit file.
  1326→- User will manually register product files to track.
  1327→- Analytics are expected to be scoped, not organization-wide.
  1328→- User understands that the app is not real-time.

  1549→## 14. Summary
  1550→
  1551→This product helps solve a core Design Ops problem: lack of visibility into where UI Kit/design system components are used.
  1552→
  1553→The product does not attempt to fully replace Figma Library Analytics or provide organization-wide automatic analytics. Instead, it provides a scoped, practical, and buildable tracker for registered files.
  1554→
  1555→The MVP should focus on:
  1556→
  1557→```txt
  1558→Source UI Kit
  1559→→ Registered Figma files
  1560→→ Manual scan
  1561→→ Component usage dashboard
  1562→→ File usage dashboard
  1563→→ Snapshot history
  1564→→ Basic change detection

  1566→
  1567→The desired user experience should feel clean, compact, and fast, taking inspiration from Figma Library Analytics for analytics concepts and Linear for interaction quality.
